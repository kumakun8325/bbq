FF6のATBバトルシステム – 詳細仕様
1. ATBゲージの仕組みと速度計算式
FF6ではバトル中、グローバルな戦闘時間カウンタが約30回/秒（1ティック≈0.033秒）で進みます
rpglegion.com
。このティック毎に、各キャラ・敵のATBゲージが以下の式で増加します
rpglegion.com
rpglegion.com
（いずれもSNES/PS版に基づく）。
# プレイヤーキャラの場合:
ATB増加量 = (96 * (素早さ + 20)) / 16     # 通常時
ATB増加量 = (126 * (素早さ + 20)) / 16    # ヘイスト時
ATB増加量 = (48  * (素早さ + 20)) / 16    # スロウ時

# 敵キャラの場合 (バトルスピードが影響):
ATB増加量 = ((96  * (素早さ + 20)) * (255 - (BatSpeed-1)*24)) / 16   # 通常時
ATB増加量 = ((126 * (素早さ + 20)) * (255 - (BatSpeed-1)*24)) / 16   # ヘイスト時
ATB増加量 = ((48  * (素早さ + 20)) * (255 - (BatSpeed-1)*24)) / 16   # スロウ時
ここで 素早さ はキャラ・敵固有のステータス値、BatSpeed はコンフィグの「バトルスピード」（1：速い～3：遅い）です
rpglegion.com
。
最終的なATBゲージ値は16ビット（0～65535）で管理され、65536到達で行動可能となります
rpglegion.com
gamefaqs.gamespot.com
。
注意点: 素早さ+20 は1バイト（0～255）に収まる必要があるため、素早さが235を超えるとオーバーフローしてATBゲージが極端に遅くなります
rpglegion.com
。
これらの計算により、素早さが高いほどゲージ増加量が大きく、行動までの時間が短くなります。上記式から、ヘイストは通常時比で約1.3125倍（+31%）に、スロウは約0.5倍に変化します。
2. バトルのリアルタイム性（ウェイト／アクティブモードの違い）
FF6ではアクティブモードとウェイトモードを切り替えられます。アクティブモードでは常に時間が進行し、プレイヤーが入力中でも他のキャラ・敵のATBゲージは増加し続けます。一方、ウェイトモードではメニューや魔法・アイテム選択中に時間が停止し、ATBゲージの増加も一時中断します
finalfantasy.fandom.com
rpglegion.com
。
アクティブモード: 戦闘時間カウンタは停止せず、アニメーション中やコマンド選択中も継続して増加します
rpglegion.com
。
ウェイトモード: プレイヤーが魔法・アイテムなどの対象選択をしている間は戦闘時間カウンタが進まず、全てのATBゲージ増加も止まります
finalfantasy.fandom.com
rpglegion.com
。
「バトルスピード」設定は戦闘時間の進み速さを決め、値を小さくするほど待機時間が短くなります
wikiwiki.jp
。SNES/PS版ではこの「バトルスピード」は敵側のATB速度にのみ影響し、味方のゲージには影響しません
wikiwiki.jp
reddit.com
（のちにPixel Remaster版で仕様変更あり
wikiwiki.jp
）。
3. 行動順とATBゲージのリセットタイミング
ATBゲージが満タンになると行動可能となり、プレイヤーはコマンド入力を行います。入力後、チャージタイム（技や魔法に応じて固定の待機時間）が経過してからコマンドが実行され、その直後にそのキャラのATBゲージは**消費され（リセットされ）**再度0から蓄積が始まります
wikiwiki.jp
finalfantasy.fandom.com
。つまり、各行動後に新たな待機時間が設定され、素早さ等に従ってゲージを再充填します。
行動順: 戦闘中はATBゲージが増えた順に行動が回ります。FF6以降は行動順を遅らせて他キャラを先に行動させる「順番送り（パス）」が可能で、FF6では前送り・後送りの選択も行えます
wikiwiki.jp
（Xボタンで前、Yボタンで後送り）。
ゲージ消費: 行動後のATBゲージは完全に消費され、次の行動に備えます
finalfantasy.fandom.com
。各行動には固有の待機時間があり、強力技（例：大魔法や連続魔）は長い待機時間を要し、かけたキャラが後回しになることもあります。一方、ジャンプやトランスは待機時間が短く、先に発動できる場合があります
wikiwiki.jp
。
先制（戦闘開始時）: 戦闘開始時はATBゲージ初期値がキャラ毎にランダムに設定されるため、素早さが低いキャラが先に行動できる場合もあります
wikiwiki.jp
。
4. ステータス異常や装備がATBに与える影響
ヘイスト (Haste): ATBゲージの蓄積速度が約31%向上します
guides.gamercorner.net
。毒やリジェネの周期も短くなります。
スロウ (Slow): ヘイストの逆効果で、蓄積速度が約半分になります（式上では上記の1/2）。複数回重複することはありません。
ストップ (Stop): 対象のATBゲージが完全に停止し、行動できなくなります
guides.gamercorner.net
。通常の毒などとは異なり、時間経過も止まるため耐久時間は長く感じられます。
その他の異常: 睡眠状態中もATBは進まず（事実上行動不能）、マヒや衰弱などはFF6には存在しません。
装備: 素早さを上げる装備（例：スプリントシューズや疾風の指輪など）で素早さが増加すれば、上記式に従ってATB蓄積速度も速くなります。なお、スクウェア作品固有の特殊装備によるゲージ操作効果はFF6にはありません。
5. 魔法・技使用中の時間停止や行動中断の仕様
通常、アクティブモード中は魔法詠唱や技の発動アニメーション中も時間が流れ、ATBゲージは増え続けます。しかし、例外として召喚魔法の発動演出中のみ時間が停止します
wikiwiki.jp
。これは召喚発動時にターゲット指定が行えないため、ゲージ停止が必須の仕様です。その他の魔法・技（詠唱魔法、特殊技、召喚以外の召喚獣技など）ではATBは進行し続けるため、発動中に敵が行動することもあります。また、ウェイトモード中はもちろん魔法・アイテム選択中の時間が止まります。行動の途中で被弾して妨害される「詠唱キャンセル」の概念はFF6にはなく、詠唱失敗や詠唱加速も発生しません。
6. 敵と味方それぞれのターン処理方法と同期の仕組み
FF6では味方と敵は同一の時間進行システムで処理されます。前述のように各ティックで全員のATBゲージが同時に増加し、どちらかのゲージが満タンになれば即座に行動できます
rpglegion.com
wikiwiki.jp
。敵は満タンになるとAIが即行動を選択するのに対し、味方はプレイヤーがコマンドを選ぶ必要があります。バトルスピード設定は敵側のみのゲージ速度に影響し、味方は常に同じ増加速度です
wikiwiki.jp
reddit.com
。攻撃アニメーション中も時間が進むため、味方の入力遅延がある程度許容されており（FF4/5のような大きな遅延はない）、多数のコマンドを連続で入力して戦闘中にキューイングすることも可能です（ただし入力中は他行動がどんどん進みます）。
ウェイトモード: プレイヤーがコマンド選択中（魔法/アイテムリストやターゲット選択）には戦闘時間カウンタが止まり、敵のATBも進みません
rpglegion.com
。
同期性: 敵味方とも16ティック毎に「内部時刻カウンタ」が更新され、これで毒・スリープ・反射などの持続時間やポイズン/リジェネダメージなどが処理されます
rpglegion.com
rpglegion.com
。これにより全体で同期した形でステータス効果が発生します。
先制/後手: バトル開始時の初期ATBはキャラ毎にランダムに決まるため、意図せず低速キャラが先に動く場合があります
wikiwiki.jp
。また、パーティ全員のゲージが満タンなら自由に行動順を決められるため、状況に応じて敵より味方を優先することも可能です。
7. その他のFF6特有のATB挙動や技術的制約
“確定行動スキップ”バグ: 敵のATBゲージが満タン状態で味方のカウンターを発動させると、敵のATBゲージが0にリセットされる不具合があります
wikiwiki.jp
。これによりボス戦で特定行動をキャンセルするテクニックが存在します。
ウィンドウ調整: 高速バトルでは演出中にターンが回ってしまう問題を逆手に取り、ウェイトモードで味方行動時にメニューを開き敵行動時に閉じることで、敵のターンをほぼ無駄にできる**「ウィンドウ調整」**がFF6から可能です
wikiwiki.jp
。
バトルスピードの挙動（リマスター版）: 2022年のピクセルリマスター版では仕様が変更され、バトルスピード設定が味方にも影響するようになりました
wikiwiki.jp
。つまり「遅い⇔速い」で味方ATB蓄積速度も目に見えて変わります。
技術的制約: ATBゲージは内部的に0～65535の16ビット値で管理されます。65536以上になると自動的に0に折り返すため、理論上一瞬で行動可能になることはなく、必ず65536到達で選択タイミングが確定します
rpglegion.com
。また、旧FFシリーズ同様、FF6でもATBはスクウェアの特許技術（Action Time Battle）によるもので、システム設計はハード/ソフト両面で最適化されています。
